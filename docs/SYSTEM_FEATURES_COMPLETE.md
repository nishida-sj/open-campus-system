## システム概要

オープンキャンパスの申込受付から参加確定、通知送信までを一元管理するWebアプリケーション。

### 対象ユーザー
- **申込者**: 中学生・高校生・保護者
- **管理者**: 学校の広報担当者・事務局スタッフ

### 主要な価値提供
- 24時間365日のオンライン申込受付
- 申込から確定までの効率的なワークフロー
- LINE連携による即座の通知配信
- CSV一括処理による大幅な工数削減
- マルチテナント対応による複数校の一元管理

## 申込者向け機能

### 1. イベント一覧表示

**URL**: `/`（トップページ）

**機能**:
- 公開中のオープンキャンパスイベント一覧を表示

### 2. 申込フォーム

**URL**: `/apply`

**機能**:

**入力項目**:

#### 基本情報
- 氏名（必須）
- ふりがな（任意）

- 電話番号（必須）

#### 学校情報
- 学校名（必須）
- 学校種別（高校/中学校/その他、任意）
- 学年（必須）

#### 参加情報
- イベント選択（必須）
- 参加希望日程（必須、複数選択可）
- 参加コース（コース設定がある場合、必須）

**バリデーション**:
- メールアドレス形式チェック
- 電話番号形式チェック
- 定員超過チェック
- 必須項目入力チェック
- 重複申込チェック

### 3. LINE連携機能（申込者側）

**機能**:

### 5. CSV一括確定機能

**概要**:
CSVファイルを使って、複数の申込者を一括で確定できる機能。100名規模の申込者でも効率的に処理可能。

**効率化**:
- 従来の方法: 100名 → 約10〜15分
- CSV一括確定: 100名 → 約2〜3分（約80%削減）

**使い方**:
1. **CSVテンプレートダウンロード**: 未確定申込者のCSVをダウンロード
2. **Excelで編集**: 「確定」列に「○」を入力（確定したい申込者のみ）
3. **CSVアップロード**: 編集したCSVファイルをアップロード
4. **処理結果確認**: 成功/失敗/スキップの詳細レポート

**機能詳細**:
- BOM付きUTF-8エンコーディング（Excel互換）
- 複数の日付形式に対応（YYYY-MM-DD、YYYY/MM/DD、YYYY年MM月DD日）
- 行ごとの個別処理（エラーがあっても他の行は成功）
- 詳細なエラーメッセージ（行番号 + 原因）
- 使い方ガイドモーダル（?ボタンで表示）

**バリデーション**:
- ヘッダー行の検証
- 申込者ID・日程・コースの存在チェック
- 複数日参加ルールの適用
- 選択日程の検証

### 6. 確定者管理

**URL**: `/admin/confirmed-list`

**機能**:
- 参加確定済み申込者の一覧表示
- LINE/メール通知の送信
- CSV出力

**フィルター機能**:
- イベント選択
- 日程フィルター（全日程 / 特定日程）

**表示情報**:
- チェックボックス（一括選択用）
- 氏名・ふりがな
- メールアドレス
- 学校名・学校種別
- 学年
- 参加日
- 参加コース
- LINE送信日（送信済みの場合）
- メール送信日（送信済みの場合）

**アクション**:

1. **LINE通知送信**:
   - 選択した確定者にLINE通知を一括送信
   - LINE連携済みの申込者のみに送信
   - 送信結果を個別に表示
   - 送信履歴を自動記録

2. **メール通知送信**:
   - 選択した確定者にメール通知を一括送信
   - 全ての確定者に送信可能
   - 送信結果を個別に表示
   - 送信履歴を自動記録

3. **CSV出力**:
   - フィルター条件に応じた確定者リストをCSV形式でダウンロード
   - Excel対応（BOM付きUTF-8）
   - ファイル名: `確定者一覧_[イベント名]_[日程]_[日付].csv`

**CSV出力項目**:
- 氏名、よみがな
- 学校名、学校種別、学年
- 参加日、コース
- メールアドレス、電話番号
- 確定日時

### 7. メールサーバー設定

**URL**: `/admin/email-settings`

## AI自動応答機能（オプション）

### 概要

申込者からLINE公式アカウントに寄せられる質問に、OpenAI（ChatGPT）が24時間365日自動で回答。

### 主要機能

#### 1. AI自動応答

**動作フロー**:
1. 申込者がLINEで質問を送信
2. Webhook経由でシステムサーバーが受信
3. データベースからカスタムプロンプトと学校情報を取得
4. OpenAI APIにプロンプトと質問を送信
5. AI回答を生成
6. LINEで申込者に回答を返信

**使用AI**: OpenAI GPT-4o mini
**応答速度**: 約2-5秒
**精度**: 95%以上（プロンプト調整により向上）

---

#### 2. プロンプトカスタマイズ機能

**管理画面から自由に編集可能**

**カスタマイズできる要素**:
- **学校基本情報**: 所在地、電話番号、アクセス方法、駐車場情報
- **イベント詳細**: 開催日時、持ち物、服装、タイムスケジュール、注意事項
- **回答トーン設定**: 丁寧語・敬語レベル、親しみやすさ、絵文字使用の有無

**管理画面機能**:
- プロンプト編集（テキストエリア）
- テスト実行（動作確認）
- 保存

**高度な設定**:
- NGワード設定（回答に含めたくない語句）
- 回答文字数制限

**プロンプト設定例**:
```
あなたは伊勢学園高等学校の広報担当者です。
丁寧で親しみやすい口調で、オープンキャンパスに関する質問に答えてください。
持ち物は筆記用具と上履きです。
駐車場は校舎裏に50台分ございます。
回答は150文字以内で簡潔にまとめてください。
```

## マルチテナント機能

### 概要

1つのシステムで複数の学校・組織を管理できるマルチテナントアーキテクチャ。

### 主要機能

#### 1. テナント管理

**機能**:
- 複数校の一元管理
- 各校ごとの独立したデータベーススキーマ
- テナント識別（サブドメイン方式 or パス方式）

**識別方法**:
- サブドメイン: `a-school.system.com` / `b-school.system.com`
- パス: `system.com/a-school` / `system.com/b-school`

---

#### 2. データ完全分離

**セキュリティ**:
- 各校のデータは独立したスキーマで物理的に分離
- 他校のデータには一切アクセス不可
- Row Level Security (RLS) による多層防御

**分離されるデータ**:
- イベント情報
- 申込者データ
- 参加確定情報
- 通知履歴
- カスタムプロンプト（AI機能使用時）

---

#### 3. コスト効果

**従来方式（2校の場合）**:
- 伊勢学園高等学校: Vercel契約1つ + Supabase契約1つ
- 伊勢保健衛生専門学校: Vercel契約1つ + Supabase契約1つ
- 合計: 4契約

**マルチテナント方式**:
- 2校共通契約: Vercel契約1つ + Supabase契約1つ（2スキーマ管理）
- 合計: 2契約
- **約40%のコスト削減**

---

#### 4. カスタマイズ対応

**各校ごとにカスタマイズ可能**:
- ロゴ・カラーテーマ
- イベント設定
- コース設定
- 確定者案内メッセージ
- AIプロンプト（AIオプション使用時）

## セキュリティ機能

### 1. 認証・認可

- **管理画面**: パスワード認証必須

- **未認証アクセス**: 自動的にログイン画面へリダイレクト

- Reactの自動エスケープ機能
- ユーザー入力をHTMLとしてレンダリングしない

### 5. Row Level Security (RLS)

**マルチテナント環境での追加セキュリティ**:
- 各テーブルにRLSを設定
- テナント間のデータ漏洩を完全防止
- 多層防御によるセキュリティ強化

### 2. データベース関数

#### increment_visit_count
指定された日程の申込カウントを+1する

```sql
CREATE OR REPLACE FUNCTION increment_visit_count(date_id UUID)
RETURNS VOID AS $$
BEGIN
  UPDATE open_campus_dates
  SET current_count = current_count + 1
  WHERE id = date_id;
END;
$$ LANGUAGE plpgsql;
```

#### decrement_visit_count
指定された日程の申込カウントを-1する

```sql
CREATE OR REPLACE FUNCTION decrement_visit_count(date_id UUID)
RETURNS VOID AS $$
BEGIN
  UPDATE open_campus_dates
  SET current_count = GREATEST(current_count - 1, 0)
  WHERE id = date_id;
END;
$$ LANGUAGE plpgsql;
```

- **Next.js**: 16.0.7（App Router）
- **React**: 19.2.1
- **TypeScript**: 5.x
- **Tailwind CSS**: 4.x
- **日付処理**: date-fns 4.1.0
- **バリデーション**: Zod 4.1.12

---

### バックエンド

- **Next.js API Routes**: サーバーレスAPI
- **Supabase**: PostgreSQLデータベース
- **LINE Messaging API**: @line/bot-sdk 10.5.0
- **OpenAI API**: openai 6.9.1（AIオプション）
- **Nodemailer**: 7.0.10（メール送信）

---

### インフラ

- **デプロイ**: Vercel
- **データベース**: Supabase (PostgreSQL)
- **ストレージ**: Supabase Storage（画像等）
- **認証**: 環境変数ベース（管理画面）

---

### セキュリティ

- **HTTPS**: 必須
- **環境変数管理**: Vercel Environment Variables
- **セッション管理**: SessionStorage
- **データ暗号化**: AES-256-GCM（マルチテナント環境）

---

## パフォーマンス最適化

### 1. データベース

- **インデックス**: 外部キー、検索頻度の高いカラムにインデックス設定
- **N+1問題回避**: JOIN または適切なSELECT戦略
- **カウント最適化**: PostgreSQL関数による効率的な集計

---

### 2. フロントエンド

- **コード分割**: Next.js動的インポート
- **画像最適化**: Next.js Image コンポーネント使用
- **静的生成**: 可能な部分はSSG（Static Site Generation）

---

### 3. API

- **並列処理**: Promise.allで複数クエリの同時実行
- **キャッシング**: 適切なキャッシュ戦略（マルチテナント環境）
- **レスポンス最適化**: 必要なデータのみ取得

---

## エラーハンドリング

### 1. フロントエンド

- ユーザーフレンドリーなエラーメッセージ
- 読み込み状態の明示（ローディングインジケーター）
- フォールバック UI（エラー時の代替表示）

---

### 2. バックエンド

- 詳細なエラーログ（console.error で記録）
- 適切なHTTPステータスコード（400, 404, 500 など）
- エラーレスポンス（JSON形式で詳細情報を返却）

---

### 3. データベース

- トランザクション（データ整合性の保証）
- ロールバック（エラー時の自動ロールバック）
- 制約違反（適切なエラーメッセージ）

---

## 運用・保守

### 1. ログ管理

- **申込ログ**: `application_logs`テーブル
- **通知履歴**: `notification_logs`テーブル
- **監査ログ**: `audit_logs`テーブル（マルチテナント環境）

---

### 2. バックアップ

- **Supabase自動バックアップ**: 日次
- **Point-in-Time Recovery (PITR)**: 有効化推奨
- **テナントごとのバックアップ**: 個別対応可能（マルチテナント環境）

---

### 3. モニタリング

- **Vercel Analytics**: パフォーマンス監視
- **Supabase Dashboard**: データベース監視
- **エラー通知**: Vercel統合機能
